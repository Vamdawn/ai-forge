# Role Playbook

根据项目情况定义审查角色，而不是照搬固定头衔。目标是让多个 SubAgent 形成互补视角。

## 选择方法

先回答四个问题：

1. 本次改动主要在需求实现、架构调整、数据模型、基础设施还是前端交互？
2. 代码范围是单模块、小型跨模块，还是跨层级系统改造？
3. 上线风险主要来自正确性、数据一致性、性能、安全，还是可维护性？
4. 历史上最常复发的问题是什么？

再从下面角色模式里选 3 到 5 个，必要时合并或改名。

## 推荐角色模式

### 1. 设计一致性审查者

适用于所有实现。重点检查：

- 是否严格按实现方案和验收标准落地
- 是否擅自修改设计或遗漏功能点
- 是否存在与设计文档冲突的实现

### 2. 架构与边界审查者

适用于跨模块、跨分层或重构类改动。重点检查：

- 模块职责是否清晰
- 分层和依赖方向是否正确
- 是否引入高耦合、隐式共享状态或难扩展结构

### 3. 正确性与健壮性审查者

适用于核心业务逻辑、异常流、并发、任务编排。重点检查：

- 边界条件
- 错误处理
- 状态转换正确性
- 潜在 panic、crash、死循环、竞态

### 4. 数据与安全审查者

适用于数据库、权限、外部接口、输入处理、命令执行。重点检查：

- 数据一致性和迁移风险
- 注入、越权、敏感信息泄露
- 不可信输入处理

### 5. 性能与资源审查者

适用于查询密集、批处理、网络/IO 密集和高频路径。重点检查：

- N+1 查询
- 重复计算
- 无效 IO 或网络调用
- 资源泄露与缓存策略

### 6. 测试与工程规范审查者

适用于所有可上线改动。重点检查：

- 关键路径测试覆盖
- 测试稳定性与可读性
- 调试代码、死代码、过时注释、lint/format 问题

### 7. 历史问题回归审查者

适用于有较多历史审查记录的项目。重点检查：

- 历史问题是否重复出现
- 旧问题是否真正修复
- 是否以新形式回归

## 组合建议

- 单模块功能开发：设计一致性 + 正确性与健壮性 + 测试与工程规范
- 跨模块重构：设计一致性 + 架构与边界 + 正确性与健壮性 + 测试与工程规范
- 含数据库或迁移：设计一致性 + 数据与安全 + 正确性与健壮性 + 测试与工程规范
- 高性能路径：设计一致性 + 性能与资源 + 正确性与健壮性 + 测试与工程规范
- 大型系统改造：设计一致性 + 架构与边界 + 正确性与健壮性 + 性能与资源 + 历史问题回归

## 角色提示

- 角色名可以按项目重命名，例如“订单域一致性审查者”“索引与检索性能审查者”。
- 每个角色只负责自己的主视角，但可以记录与主视角强相关的次要问题。
- 角色之间允许交叉发现问题，但主代理在合并时必须去重。
