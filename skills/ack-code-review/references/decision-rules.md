# Review Issue Decision Rules

这个参考文件用于帮助主 skill 判断单条审查意见是否应被采纳，以及在输出中如何解释该判断。

## 目标

你不是审查意见的抄写员。你的职责是判断这条意见在当前项目上下文里是否成立，并在成立时把问题修复到真实可运行的状态。

## 单条 issue 的判断模板

对每条审查意见，至少在心里完成以下判断：

1. 这条意见指出的现象是否真实存在
2. 它影响的是正确性、稳定性、可维护性，还是仅仅是风格偏好
3. 它是否和实现方案、架构约束、现有模块边界一致
4. 如果采纳，最小但完整的修复范围是什么
5. 如果不采纳，证据是什么

## 推荐的内部整理格式

可按以下结构整理，但不要求原样输出：

```text
[Issue]
- title:
- status: 采纳 | 不采纳 | 待确认
- reason:
- evidence:
- impacted_files:
- fix_plan:
```

## 采纳的典型信号

- 能复现 bug 或明显错误
- 审查指出缺失的边界处理，而代码确实遗漏
- 审查指出与方案不一致，而方案中有明确约束
- 审查指出测试缺口，且缺口覆盖关键路径
- 审查指出结构问题，且已经造成重复逻辑、职责错位或高耦合

## 不采纳的典型信号

- 审查描述和代码事实不符
- 建议违反当前项目既定模式，且没有足够收益
- 只是 reviewer 的个体偏好，没有带来明确质量提升
- 审查建议解决的是假问题，采纳反而会引入复杂度或回归

## 待确认的典型信号

- 需要业务决策才能判断
- 需要外部系统行为或线上事实作为前提
- 审查报告缺少定位，代码中存在多个候选解释

`待确认` 不是默认兜底。只有在缺证据且风险高时才保留。

## 修复策略

当你决定采纳后：

1. 先找根因，不要只修表象
2. 优先做局部、完整、可验证的增量修改
3. 必要时补测试，证明修复成立
4. 任何结构调整都要能解释“为什么这是最小必要变更”

## 输出要求

对用户汇报时，确保每条 issue 至少能回答：

- 采纳了吗
- 为什么
- 改了哪里
- 如何验证

若不采纳，也要给出简洁但可审查的技术依据，而不是一句“我认为没必要”。
