# retrospect-session 结构治理优化设计

## 概述

本设计针对 `skills/retrospect-session/SKILL.md` 的执行质量进行优化，重点解决“历史规则很多时，skill 不会重整体系，只会机械追加”的问题。

优化目标不是提升触发率，也不是先补 eval，而是把该 skill 从“把本次会话经验写进规则文件”的执行清单，升级为“先盘点规则体系、再设计目标结构、再重写落盘并同步索引”的治理工作流。

## 问题定义

当前 skill 已具备较完整的模板、同步要求和输出格式，但在以下关键点上仍然偏弱：

- 缺少“何时必须进入体系重整”的明确判断条件。
- 缺少“新增 / 合并 / 拆分 / 迁移 / 废弃”之间的决策准则。
- 虽然强调去重和冲突处理，但没有把“规则版图盘点”和“目标结构设计”前置为默认动作。
- 容易把会话中的具体事件直接写成规则，导致规则库逐渐变成经验堆积，而不是稳定的信息架构。

这会导致 skill 在面对历史规则较多、主题边界漂移、索引失真的仓库时，继续沿用增量追加路径，进一步放大重复和冲突。

## 设计目标

1. 让该 skill 默认先判断当前任务是“增量更新”还是“体系重整”。
2. 让模型在写入前先完成规则盘点，并设计目标结构。
3. 让“重写、合并、拆分、迁移、废弃”成为明确授权动作，而不是只会追加。
4. 让 `AGENTS.md` / `CLAUDE.md` 的索引同步建立在整理后的结构之上。
5. 让最终汇报体现“规则体系如何被治理”，而不只是“改了哪些文件”。

## 非目标

- 本轮不优化 skill 的 description 触发边界。
- 本轮不引入 `evals/`、benchmark 或基线测试流程。
- 本轮不改变 `docs/rules/*.md` 与项目级提示词的基本模板格式。

## 方案选型

### 方案 A：轻量增强型

仅在现有流程中补充若干判断准则，保留整体线性 checklist。

- 优点：改动最小，落地风险低。
- 缺点：仍容易退化成“读一遍现状，然后开始局部追加”。

### 方案 B：重整优先型

把 skill 主流程改成“先盘点现状，再设计目标结构，再统一落盘与同步索引”。

- 优点：直接瞄准当前核心问题。
- 缺点：流程更长，需要更明确的中间判断点。

### 方案 C：双层治理型

区分“增量更新”和“体系重整”两种模式，根据结构性症状切换。

- 优点：更符合真实项目，不会每次都过度重构。
- 缺点：如果切换条件不清晰，执行时容易摇摆。

### 推荐方案

采用 `B + C` 组合：

- 主体流程采用“重整优先型”。
- 在流程前增加模式判断，只有满足严格条件时才允许走增量更新。

这样既能把“体系治理”前置成默认动作，也能避免简单场景下的过度重构。

## 核心设计

## Operating Modes

为 skill 增加两种明确模式：

### 1. Incremental Update

仅在以下条件同时成立时允许进入：

- 只涉及 `1-2` 个 `docs/rules/*.md` 文件。
- 未发现跨文件重复或职责冲突。
- 现有主题边界仍清晰。
- `AGENTS.md` / `CLAUDE.md` 只需小幅同步，不需要重写索引摘要结构。

### 2. System Restructure

满足任一条件即进入：

- 本次改动涉及 `3` 个及以上规则文件。
- 存在同义规则跨文件重复。
- 某个规则文件承载多个主题，职责混杂。
- 现有主题边界已无法容纳新教训而不引入漂移。
- 项目级索引描述与实际规则职责不一致。
- 本次需要解释“哪些规则应该保留、迁移、合并或删除”，而不是仅新增内容。

默认优先判断是否需要 `System Restructure`，而不是默认走增量更新。

## Workflow

将当前线性 checklist 改为以下治理流程：

### 1. Assess Current System

- 回顾当前会话中的目标、返工点、失败模式和有效做法。
- 读取 `docs/rules/**/*.md`、`AGENTS.md`、`CLAUDE.md`。
- 判断当前任务属于增量更新还是体系重整。

### 2. Build Rule Inventory

在真正写文件前，先建立“规则版图”：

- 现有主题文件有哪些。
- 每个文件的职责边界是什么。
- 哪些规则重复、冲突、漂移或放错位置。
- 项目级索引是否真实反映当前结构。

这一步的目标不是列文件名，而是建立结构判断基础。

### 3. Design Target Structure

先设计整理后的目标结构，再开始落盘：

- 保留哪些文件。
- 新增哪些文件。
- 合并哪些文件。
- 拆分哪些文件。
- 迁移哪些规则。
- 废弃哪些已被覆盖的旧规则。

只有目标结构明确后，才进入写入步骤。

### 4. Apply Rewrite / Merge / Split / Retire

把动作边界写清楚：

- `Rewrite`：当文件职责已混杂、规则顺序失真或局部 patch 无法恢复结构时，整体重写该主题文件。
- `Merge`：当两个文件约束同一类决策，只是历史表述不同，应合并为一个更稳定主题。
- `Split`：当一个文件同时覆盖多个独立场景，继续共存会降低可检索性时，应拆分。
- `Retire`：当旧规则已被更通用、更可执行的规则完全覆盖时，应删除旧条目，而不是并列保留。
- `Migrate`：当规则仍有效但归属主题不再合适时，应迁移到更合适的文件，并在变更记录说明。

默认优先“重整后覆盖写入”，而不是在原文件上累积拼补。

### 5. Sync Project Prompts

在结构整理完成后，再同步 `AGENTS.md` / `CLAUDE.md`：

- `Rules Index` 必须反映当前真实文件结构。
- 如果规则文件职责变化，索引摘要必须重写，不能沿用旧表述。
- 若两份项目级提示词冲突，以更具体、更贴近当前结构的一版为基准统一。

### 6. Run Consistency Checks

最终检查聚焦治理结果，而非仅检查语法：

- 是否仍有跨文件重复。
- 是否仍有职责边界不清。
- `Rules Index` 是否与 `docs/rules/*.md` 一致。
- 是否把一次性经验错误地沉淀成长期规则。

### 7. Report Governance Changes

最终汇报需要回答：

- 本次是增量更新还是体系重整。
- 为什么做出这个判断。
- 哪些规则被合并、拆分、迁移、废弃。
- 当前结构相比之前消除了哪些重复和冲突。

## 抽象标准

新增一节明确限制经验沉淀方式：

- 不要把本次会话中的具体事件原样写成长期规则。
- 先识别它对应的可重复失败模式，再提炼为“触发条件 + 执行动作 + 验证方式”。
- 如果某条经验只适用于当前一次会话，不应写入长期规则库，只应体现在变更记录或对用户汇报中。

该标准用于阻止规则库退化成经验日志。

## SKILL.md 结构重组建议

建议将现有 `SKILL.md` 重组为以下骨架：

1. `Overview`
2. `Goals`
3. `Mandatory Outcomes`
4. `Operating Modes`
5. `Restructure Triggers`
6. `Workflow`
7. `Abstraction Standard`
8. `Rule Writing Standard`
9. `docs/rules Template`
10. `AGENTS.md / CLAUDE.md Template`
11. `Output Format`
12. `Edge Cases`

其中：

- 保留现有模板、输出格式与异常处理章节。
- 改写当前“执行流程”为治理导向流程。
- 新增模式判断、规则盘点、目标结构设计、动作语义和抽象标准。

## 成功标准

完成优化后，该 skill 应该能稳定表现出以下行为：

- 在存在重复、冲突、职责漂移时，先做结构盘点，而不是直接追加。
- 能明确判断何时应整体重写某个主题文件。
- 允许删除、迁移、合并旧规则，而不是默认保留所有历史形态。
- 索引同步建立在整理后的结构之上，而不是成为结尾附带步骤。
- 输出给用户的结果能解释“规则体系如何被治理”。

## 风险与权衡

- 流程变长：治理型流程比纯追加更重，但这是为了解决结构劣化问题。
- 模型可能过度重构：因此需要保留严格的增量更新条件，避免每次都进入大规模整理。
- 文档变长：需要控制 `SKILL.md` 仍保持可读，避免因治理概念过多而失焦。

## 后续实现建议

后续实现应按小步拆分，避免一次改动过大：

1. 先只重构 `SKILL.md` 的主流程和章节结构。
2. 再补入模式判断、重整触发条件和动作语义。
3. 最后再精修模板、输出格式与措辞，确保整份 skill 读起来连贯。
